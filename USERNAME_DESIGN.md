# Unique usernames for users â€” design report

A unique handle per user, generated by the server, overridable by plugins, changeable by
players â€” with uniqueness and format owned by core.

- **Scope:** `game_server_core` + web serialization
- **Status:** awaiting review before implementation

## 1. The model: two fields, two jobs

`username` is a new **unique, stable handle** â€” used for friend-add, search, mentions, URLs.
The existing `display_name` stays the **non-unique pretty name** shown on scoreboards and in
UI. Two Nicos in one match is fine; the scoreboard shows `display_name`, so the "9999 Nicos"
problem never reaches the player.

This is where the industry landed: Discord abandoned `name#1234` in 2023 (players couldn't
remember discriminators, and popular names really did exhaust 9999); Steam has always split
unique account name from non-unique persona name; Riot's tagline is user-chosen and 3â€“5
chars, not a sequential counter.

Games that don't want handles simply never expose the field â€” it exists, it's unique, it
costs nothing.

## 2. The rule

**Format â€” enforced by core, not overridable by hooks:**

```
^[a-z0-9](?:[._-]?[a-z0-9])*$        # plus length 3â€“32
```

- **Charset:** `aâ€“z`, `0â€“9`, and the separators `.` `_` `-` (ASCII only â€” Unicode lives in
  `display_name`)
- **Shape:** starts and ends alphanumeric; no consecutive separators (the regex encodes all
  three constraints)
- **Length:** 3â€“32, via `GameServer.Limits` (`:min_username` / `:max_username`) so games can
  tighten it
- **Case:** input is lowercased on cast (Discord-style). Stored lowercase â†’ uniqueness is a
  plain unique index, no `lower()` tricks, no case-impersonation

| Input        | Result       | Why                                |
| ------------ | ------------ | ---------------------------------- |
| `Dragos.Dev` | `dragos.dev` | lowercased on cast                 |
| `sheep-4821` | `sheep-4821` | generated form is itself valid     |
| `ni`         | rejected     | below min length                   |
| `nico..test` | rejected     | consecutive separators             |
| `-nico`      | rejected     | leading separator                  |
| `nicÃ¶`       | rejected     | non-ASCII â€” belongs in display_name |

Reserved names (`admin`, profanity, brand names) are deliberately **not** core's job â€” a
plugin rejects them in the hook. Core guarantees shape and uniqueness only.

## 3. Generation

Every new user gets a generated username before any hook runs. The base depends on what we
safely know:

| Signup source                                  | Base                     | Example       |
| ---------------------------------------------- | ------------------------ | ------------- |
| OAuth (Discord, Google, Apple, Steam, Facebook) | slugified provider name  | `dragos-4821` |
| Email / magic link                             | random word from list    | `sheep-4821`  |
| Device (guest)                                 | random word from list    | `falcon-0392` |

**Never derived from the email local-part.** With only a handful of big mail providers,
`nico.test` as a username lets anyone guess `nico.test@gmail.com` in a few tries â€” a real
enumeration/spam vector.

- **Slugify:** best-effort ASCII transliteration ("DragoÈ™" â†’ `dragos`), strip to charset,
  collapse separators, truncate to fit suffix; if fewer than 3 chars survive, fall back to
  the wordlist.
- **Suffix:** 4 random digits â€” random, not sequential, so there's no counter to exhaust.
  On collision retry with a fresh number (3 attempts), then widen to 6 digits. With a
  ~200-word list this space is effectively inexhaustible.
- **Uniqueness race:** the DB unique index is the referee; insert conflicts feed the retry
  loop.

## 4. Hook: `before_user_register`

A new pipeline hook, same family as `before_user_update` â€” plugins already compose via
`run_before_pipeline` (`reduce_while` chaining), so "what if two hooks extend this?" is
already answered: each plugin receives the previous plugin's attrs, in plugin order; the
last write wins; any `{:error, reason}` halts registration.

```elixir
@callback before_user_register(attrs :: map()) :: {:ok, map()} | {:error, term()}
# attrs arrive string-keyed and already contain the generated "username";
# provider is inferable from keys ("google_id", "device_id", ...)
```

Flow:

1. **Core generates** a default username into attrs (per the table above).
2. **Pipeline runs** â€” each plugin may replace the username or touch any other attr. No
   plugins configured â†’ attrs pass through untouched.
3. **Core validates** â€” lowercase, format regex, length. Non-negotiable, runs *after* the
   hooks.
4. **Insert with collision fallback** â€” generated name collides: retry with fresh suffix.
   Hook-supplied name collides or is invalid: log a warning and fall back to a generated
   name.

> **Why the fallback matters:** `find_or_create_from_*` runs during *login*. A plugin bug
> that yields a colliding or malformed username must never lock a player out â€” creation is
> resilient. The player-initiated *update* path is the opposite: strict errors
> (`:username_taken`, `:invalid_username`) so the player can pick again.

## 5. Changing it later

`Accounts.update_username(user, attrs)`, strict validation, routed through the existing
`before_user_update` pipeline â€” so each game decides its own policy (forbid entirely,
cooldown, one free changeâ€¦) without core taking a position. Plus
`Accounts.get_user_by_username/1` for lookup.

## 6. Data & migration

- Add `users.username` â†’ backfill every existing row with the generator (batched, in the
  migration) â†’ set `NOT NULL` + unique index. One deploy, no nullable-column shim, no lazy
  generate-on-login path.
- `User.t()` gains `username: String.t()`; new `username_changeset/2` holds normalization +
  validation + `unique_constraint`.

## 8. Open decisions

- **Separator set:** proposal allows all of `.` `_` `-`; generated names use `-`. Happy to
  restrict to one.
- **Max length 32:** Discord's number; games can lower via Limits. Fine?
- **Guest visibility:** device users get `falcon-0392` like everyone else â€” no special
  "guest" prefix. Agreed?
